                        1. SYSTEM OVERVIEW
================================================================================

PROJECT NAME: TransitFlow Analytics System

DESCRIPTION:
TransitFlow is an enterprise-level analytics platform designed for processing
and analyzing large-scale logistics and e-commerce data. The system handles
500,000+ records efficiently, providing real-time insights through interactive
dashboards and advanced analytics.

KEY FEATURES:
- Bulk dataset ingestion from CSV files
- Chunk-based processing (5,000 records per chunk)
- 7 advanced analytics strategies
- Interactive dashboard with visualizations
- Dead-Letter Queue (DLQ) for faulty records
- Real-time performance metrics
- Geographic and behavioral customer insights
- Revenue analysis and anomaly detection

TECHNOLOGY STACK:

Backend:
- Framework: Spring Boot 3.3.5
- Language: Java 21 (Runtime: Java 24)
- Database: H2 In-Memory Database
- ORM: Spring Data JPA / Hibernate
- Build Tool: Maven 3.9.5
- Libraries: Apache Commons CSV, Lombok

Frontend:
- Framework: Streamlit 1.29.0
- Language: Python 3.10+
- Visualization: Plotly, Altair
- HTTP Client: Requests library

                   4. DESIGN PATTERNS IMPLEMENTED

PATTERN 1: FACADE PATTERN
Location: com.transitflow.facade.TransitFlowFacade

Purpose:
Provides a simplified, unified interface to complex subsystems (DatasetService
and multiple Analytics Strategies). Shields the controller layer from complex
implementation details.

Benefits:
- Controllers call one method instead of managing multiple services
- Decouples controllers from service implementations
- Single point for cross-cutting concerns (logging, caching)
- Easy to maintain and extend

Implementation:
public class TransitFlowFacade {
    @Autowired
    private DatasetService datasetService;
    @Autowired
    private PeakDeliveryStrategy peakStrategy;
    @Autowired
    private RevenueAnalysisStrategy revenueStrategy;
    // ... 7 strategy implementations
    
    public Map<String, Object> executeAnalysis(String strategyName) {
        AnalyticsStrategy strategy = selectStrategy(strategyName);
        return strategy.execute();
    }
}

Assignment Mapping: ✅ M2 - Design Pattern Evidence

PATTERN 2: STRATEGY PATTERN
Location: com.transitflow.strategy.*

Purpose:
Encapsulates interchangeable analytics algorithms, allowing them to be selected
and executed dynamically at runtime.

Benefits:
- Each strategy is independent and testable
- Easy to add new analytics without modifying existing code
- Algorithms can be switched at runtime
- Follows Open-Closed Principle (SOLID)

Interface:
public interface AnalyticsStrategy {
    Map<String, Object> execute();
    String getStrategyName();
}

Implementations (7 strategies):
1. PeakDeliveryStrategy - Identifies peak order times (hourly, daily, weekly)
2. CustomerSegmentationStrategy - Geographic and behavioral analysis
3. DeliveryEfficiencyStrategy - Transit time and delay metrics
4. RevenueAnalysisStrategy - Financial performance tracking
5. AnomalyDetectionStrategy - Statistical outlier detection
6. SeasonalDemandStrategy - Seasonal pattern identification
7. RegionalPerformanceStrategy - Geographic performance analysis

Example Usage:
AnalyticsStrategy strategy = new RevenueAnalysisStrategy();
Map<String, Object> results = strategy.execute();

PATTERN 3: ADAPTER PATTERN
Location: com.transitflow.adapter.*

Purpose:
Converts different file formats (CSV, JSON, Excel) into standardized entity

Benefits:

public interface DatasetAdapter<T> {
    List<List<T>> parseInChunks(int chunkSize) throws IOException;
    long getTotalRecords() throws IOException;
    String getSourceIdentifier();
}

Implementation:
public class CsvDatasetAdapter<T> implements DatasetAdapter<T> {
    private Path filePath;
    private Function<CSVRecord, T> mapper;
    
    public List<List<T>> parseInChunks(int chunkSize) {
        // Parse CSV and convert to entity chunks
    }
}

Flow:
CSV File → CsvDatasetAdapter → Customer Entity → Repository → Database
JSON File → JsonDatasetAdapter → Customer Entity → Repository → Database
Excel File → ExcelDatasetAdapter → Customer Entity → Repository → Database

PATTERN 4: FACTORY PATTERN
Location: com.transitflow.factory.DatasetLoaderFactory

Purpose:
Creates appropriate DatasetAdapter based on file type, encapsulating object
creation logic.

Benefits:
- Encapsulates object creation
- Easy to add new file formats
- Decouples client from concrete classes
- Follows Open-Closed Principle (SOLID)

Implementation:
public class DatasetLoaderFactory {
    public static DatasetAdapter<Customer> createCustomerLoader(Path path) {
        String fileName = path.getFileName().toString().toLowerCase();
        
        if (fileName.endsWith(".csv")) {
            return new CsvDatasetAdapter<>(path, customerMapper());
        } else if (fileName.endsWith(".json")) {
            return new JsonDatasetAdapter<>(path);
        } else if (fileName.endsWith(".xlsx")) {
            return new ExcelDatasetAdapter<>(path);
        }
        throw new UnsupportedOperationException("Unsupported file type");
    }
    
    // Similar methods for Order, Payment, Product, Seller
}

Assignment Mapping: ✅ M2 - Design Pattern Evidence

PATTERN 5: REPOSITORY PATTERN
Location: com.transitflow.repository.*

Purpose:
Abstracts data access logic, providing a collection-like interface for
accessing domain objects.

Benefits:
- Decouples business logic from data access
- Easy to switch database implementations
- Testable with mock repositories
- Centralized query logic

Implementation:
public interface CustomerRepository extends JpaRepository<Customer, String> {
    List<Customer> findAll();
    Customer save(Customer customer);
    Long count();
    
    // Custom aggregated query
    @Query("SELECT c.customerState, COUNT(c) FROM Customer c 
            GROUP BY c.customerState")
    List<Object[]> countByState();
}

Spring Data JPA automatically implements these interfaces at runtime.

                   8. PERFORMANCE OPTIMIZATIONS
================================================================================

PROBLEM 1: N+1 QUERY PROBLEM

Before Optimization:
--------------------------------------
Code:
for (Customer customer : customers) {  // 99,441 iterations
    Long orderCount = orderRepository.countByCustomerId(customer.getId());
}

Issue:
- Makes 99,441 separate database queries
- Each query waits for round-trip to database
- Total time: 30-60+ seconds ❌

After Optimization:
--------------------------------------
Code:

PROBLEM 2: LARGE FILE PROCESSING

Solution: Chunk-Based Processing
--------------------------------------
Implementation:
- Process 5,000 records per chunk
- Memory-efficient streaming
- Prevents OutOfMemoryError
- Progress tracking per chunk

Code:
public void processDataset(DatasetAdapter<T> adapter) {
    List<List<T>> chunks = adapter.parseInChunks(5000);
    
    for (List<T> chunk : chunks) {
        chunk = removeDuplicates(chunk);
        repository.saveAll(chunk);
        // Memory released after each chunk
    }
}

Benefits:
- Handles 500,000+ records efficiently
- Constant memory usage
- Can process files larger than available RAM
- Graceful handling of errors per chunk

ISSUE 1: JAVA_HOME not found
--------------------------------------
Error Message:
"Error: JAVA_HOME not found in your environment"

Solution:
1. Verify Java installation:
   java -version
   
2. Set JAVA_HOME:
   setx JAVA_HOME "C:\Program Files\Java\jdk-24"
   
3. Restart Command Prompt
4. Verify:
   echo %JAVA_HOME%

--------------------------------------------------------------------------------

ISSUE 2: Port 8080 already in use
--------------------------------------
Error Message:
"Web server failed to start. Port 8080 was already in use."

Solution:
1. Find process using port 8080:
   netstat -ano | findstr :8080
   
2. Kill the process:
   taskkill /PID <process_id> /F
   
3. Or change port in application.properties:
   server.port=8081


--------------------------------------------------------------------------------

ISSUE 3: Frontend cannot connect to backend
--------------------------------------
Error Message:
"HTTPConnectionPool(host='localhost', port=8080): Connection refused"

Solution:
1. Verify backend is running:
   Open browser: http://localhost:8080
   
2. Check firewall settings
3. Verify CORS configuration in WebConfig.java
4. Check backend logs for errors

--------------------------------------------------------------------------------

ISSUE 4: Timeout errors on analytics
--------------------------------------
Error Message:
"Read timed out. (read timeout=30)"

Solution:
1. Increase timeout in frontend:
   response = requests.get(url, timeout=120)
   
2. Check backend performance
3. Verify database has data (run stats endpoint)
4. Optimize queries (see Performance section)

--------------------------------------------------------------------------------

ISSUE 5: CSV file not found
--------------------------------------
Error Message:
"Directory not found or invalid"

Solution:
1. Verify directory path is absolute:
   Good: C:\Users\User\Desktop\Data
   Bad: .\Data or ../Data
   
2. Check all 6 CSV files are present
3. Verify file names match exactly:
   - customers.csv
   - orders.csv
   - order_items.csv
   - payments.csv
   - products.csv
   - sellers.csv
   
4. Check file permissions

--------------------------------------------------------------------------------

ISSUE 6: OutOfMemoryError during ingestion
--------------------------------------
Error Message:
"java.lang.OutOfMemoryError: Java heap space"

Solution:
1. Increase JVM heap size:
   mvnw spring-boot:run -Dspring-boot.run.jvmArguments="-Xmx4g"
   
2. Reduce chunk size in DatasetService:
   CHUNK_SIZE = 2000; // Instead of 5000
   
3. Close other applications to free memory

--------------------------------------------------------------------------------

ISSUE 7: Streamlit module not found
--------------------------------------
Error Message:
"ModuleNotFoundError: No module named 'streamlit'"

Solution:
1. Verify Python installation:
   python --version
   
2. Install requirements:
   pip install -r requirements.txt
   
3. If using virtual environment, activate it first:
   venv\Scripts\activate
   
4. Verify installation:
   pip list | findstr streamlit

--------------------------------------------------------------------------------

ISSUE 8: Blank charts or "No data" messages
--------------------------------------
Possible Causes:
- Dataset not ingested
- Backend returned empty results
- Frontend parsing error

Solution:
1. Verify data ingestion:
   GET http://localhost:8080/api/dataset/stats
   
2. Check backend logs for errors
3. Verify API response format
4. Clear browser cache
5. Re-ingest dataset if needed

APPENDIX A: TECHNOLOGY JUSTIFICATION
--------------------------------------

Spring Boot:
Chosen for:
- Industry-standard framework
- Built-in dependency injection
- Auto-configuration
- Embedded server
- Large ecosystem

Alternatives considered: Micronaut, Quarkus
Reason for choice: Maturity, community support, extensive documentation

H2 Database:
Chosen for:
- Fast in-memory processing
- Zero configuration
- Perfect for development/demo
- Easy testing

Alternatives considered: PostgreSQL, MySQL
Reason for choice: Simplicity for academic project
Production recommendation: PostgreSQL

Streamlit:
Chosen for:
- Rapid development
- Python-based (matches data science workflow)
- Built-in visualization
- No frontend expertise required

Alternatives considered: React, Angular, Vue
Reason for choice: Speed of development, focus on functionality

JPA/Hibernate:
Chosen for:
- ORM abstraction
- Reduced boilerplate SQL
- Database independence
- Automatic schema generation

Alternatives considered: jOOQ, MyBatis
Reason for choice: Integration with Spring Data

APPENDIX B: DATASET SCHEMA
--------------------------------------

customers.csv:
- customer_id: Unique identifier (VARCHAR)
- customer_unique_id: Alternative ID (VARCHAR)
- customer_zip_code_prefix: Postal code (VARCHAR)
- customer_city: City name (VARCHAR)
- customer_state: State code (VARCHAR, 2 letters)

orders.csv:
- order_id: Unique identifier (VARCHAR)
- customer_id: Foreign key to customers (VARCHAR)
- order_status: Status (delivered, shipped, etc.)
- order_purchase_timestamp: Purchase date (YYYY-MM-DD HH:MM:SS)
- order_approved_at: Approval date (YYYY-MM-DD HH:MM:SS)
- order_delivered_carrier_date: Carrier pickup (YYYY-MM-DD HH:MM:SS)
- order_delivered_customer_date: Delivery date (YYYY-MM-DD HH:MM:SS)
- order_estimated_delivery_date: Estimated date (YYYY-MM-DD HH:MM:SS)

order_items.csv:
- order_id: Foreign key to orders (VARCHAR)
- order_item_id: Item sequence (INTEGER)
- product_id: Foreign key to products (VARCHAR)
- seller_id: Foreign key to sellers (VARCHAR)
- shipping_limit_date: Shipping deadline (YYYY-MM-DD HH:MM:SS)
- price: Item price (DOUBLE)
- freight_value: Shipping cost (DOUBLE)

payments.csv:
- order_id: Foreign key to orders (VARCHAR)
- payment_sequential: Payment sequence (INTEGER)
- payment_type: Payment method (credit_card, boleto, voucher, debit_card)
- payment_installments: Number of installments (INTEGER)
- payment_value: Payment amount (DOUBLE)

products.csv:
- product_id: Unique identifier (VARCHAR)
- product_category_name: Category (VARCHAR)
- product_name_lenght: Name length (INTEGER)
- product_description_lenght: Description length (INTEGER)
- product_photos_qty: Photo count (INTEGER)
- product_weight_g: Weight in grams (INTEGER)
- product_length_cm: Length in cm (INTEGER)
- product_height_cm: Height in cm (INTEGER)
- product_width_cm: Width in cm (INTEGER)

sellers.csv:
- seller_id: Unique identifier (VARCHAR)
- seller_zip_code_prefix: Postal code (VARCHAR)
- seller_city: City name (VARCHAR)
- seller_state: State code (VARCHAR, 2 letters)
