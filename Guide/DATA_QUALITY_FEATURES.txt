1. AUTOMATIC DUPLICATE REMOVAL
================================================================================

WHAT IT IS:
-----------
A mechanism that automatically detects and removes duplicate records during
data ingestion to ensure data integrity and prevent redundant entries.

HOW IT WORKS:
-------------
The system uses a HashSet-based approach to track unique record identifiers
during the ingestion process. When processing CSV files in chunks:

1. Each chunk is processed sequentially
2. A HashSet maintains all seen record IDs
3. Before saving, each record is checked against the HashSet
4. If ID exists → Record is skipped (duplicate)
5. If ID is new → Record is added to HashSet and saved

ALGORITHM:
----------
Input: List of records from CSV chunk
Output: List of unique records only

Process:
  FOR each chunk in CSV file:
    Initialize empty HashSet called "seenIds"
    Initialize empty List called "uniqueRecords"
    
    FOR each record in chunk:
      IF record.id NOT IN seenIds:
        ADD record.id to seenIds
        ADD record to uniqueRecords
      ELSE:
        INCREMENT duplicateCounter
    
    SAVE uniqueRecords to database

private void processCustomers(Path dirPath, Map<String, Object> result) {
    // ... file loading code ...
    
    Set<String> seenIds = new HashSet<>();  // ← Tracks unique IDs
    int duplicatesRemoved = 0;
    
    for (List<Customer> chunk : chunks) {
        List<Customer> validChunk = new ArrayList<>();
        
        for (Customer customer : chunk) {
            // Check if ID already exists
            if (!seenIds.contains(customer.getCustomerId())) {
                validChunk.add(customer);
                seenIds.add(customer.getCustomerId());  // ← Mark as seen
            } else {
                duplicatesRemoved++;  // ← Count duplicate
            }
        }
        
        customerRepository.saveAll(validChunk);  // ← Save only unique records
    }
    
    result.put("customers_duplicates", duplicatesRemoved);
}

2. DATA VALIDATION & CLEANSING
================================================================================

WHAT IT IS:
-----------
A multi-stage validation process that checks data quality and filters out
invalid, incomplete, or malformed records before database insertion.

HOW IT WORKS:
-------------
The system implements validation rules for each entity type:

STAGE 1: Null Check
  - Verify required fields are not null
  - Check primary key existence

STAGE 2: Empty String Check
  - Trim whitespace
  - Reject empty strings

STAGE 3: Data Type Validation
  - Verify numeric fields contain valid numbers
  - Validate date formats
  - Check data type consistency

STAGE 4: Business Rule Validation
  - Validate payment values > 0
  - Ensure timestamps are logical (delivery after purchase)
  - Check referential integrity

VALIDATION RULES BY ENTITY:
----------------------------

CUSTOMER:
  ✓ customerId must not be null
  ✓ customerId must not be empty (after trim)
  ✓ Optional: customerState must be 2 characters

ORDER:
  ✓ orderId must not be null
  ✓ orderId must not be empty
  ✓ orderStatus must be valid (delivered, shipped, etc.)
  ✓ orderPurchaseTimestamp must be valid date

PAYMENT:
  ✓ orderId must not be null
  ✓ paymentValue must be a valid number
  ✓ paymentValue should be >= 0
  ✓ paymentType must not be empty

private boolean isValidCustomer(Customer customer) {
    return customer != null && 
           customer.getCustomerId() != null && 
           !customer.getCustomerId().trim().isEmpty();
}

private boolean isValidOrder(Order order) {
    return order != null && 
           order.getOrderId() != null && 
           !order.getOrderId().trim().isEmpty() &&
           order.getCustomerId() != null;
}

private boolean isValidPayment(Payment payment) {
    return payment != null && 
           payment.getOrderId() != null && 
           payment.getPaymentValue() != null &&
           payment.getPaymentValue() >= 0;
}

3. DEAD-LETTER QUEUE (DLQ) FOR ERRORS
================================================================================

WHAT IT IS:
-----------
A separate database table (faulty_records) that captures and stores records
that failed processing, allowing for analysis and recovery without data loss.

WHY IT'S NEEDED:
----------------
During large-scale data ingestion:
- Some records may have malformed data
- Parsing errors might occur
- Validation failures need to be tracked
- We don't want to lose data completely
- We need to analyze patterns in bad data

HOW IT WORKS:
-------------
The DLQ implementation follows these steps:

1. Record Processing Attempt
   ↓
2. Error Detected (parsing, validation, etc.)
   ↓
3. Catch Exception
   ↓
4. Create FaultyRecord entry with:
   - Source file name
   - Raw content (if available)
   - Error message
   - Timestamp
   ↓
5. Save to faulty_records table
   ↓
6. Continue processing (fault-tolerant)

DATABASE SCHEMA:
----------------
Table: faulty_records

Columns:
  - id (BIGINT, Primary Key, Auto-increment)
  - source_file (VARCHAR) - Which CSV file contained the error
  - raw_content (VARCHAR, 2000) - The actual problematic data
  - failure_reason (VARCHAR, 1000) - Error message explaining what went wrong
  - timestamp (TIMESTAMP) - When the error occurred

-------------

4. FAULT-TOLERANT ARCHITECTURE
================================================================================

WHAT IT IS:
-----------
An architectural approach that allows the system to continue functioning
even when individual components fail or encounter errors.

PRINCIPLES APPLIED:
-------------------

1. GRACEFUL DEGRADATION
   - System continues processing when encountering bad records
   - One failed record doesn't stop entire batch
   - Partial success is acceptable

2. EXCEPTION HANDLING
   - Try-catch blocks at multiple levels
   - Errors logged but don't crash system
   - Error details captured for analysis

3. TRANSACTIONAL BOUNDARIES
   - Each chunk processed in separate transaction
   - Failed chunk doesn't rollback previous chunks
   - Database consistency maintained

4. DEFENSIVE PROGRAMMING
   - Null checks before operations
   - Validation before processing
   - Safe type conversions

FAULT-TOLERANT PATTERNS IMPLEMENTED:
-------------------------------------

PATTERN 1: Chunk-Based Processing
----------------------------------
Instead of processing all records at once:

✗ BAD (All-or-Nothing):
  Load 1,000,000 records
  Process all records
  Save all records
  → If any record fails, entire batch fails

✓ GOOD (Fault-Tolerant):
  FOR each chunk of 5,000 records:
    Load chunk
    Process chunk
    Save chunk
  → If one chunk fails, others still succeed

CODE EVIDENCE:


private static final int CHUNK_SIZE = 5000;  // ← Configurable chunk size

List<List<Customer>> chunks = adapter.parseInChunks(CHUNK_SIZE);

for (List<Customer> chunk : chunks) {
    try {
        // Process this chunk
        List<Customer> validChunk = new ArrayList<>();
        
        for (Customer customer : chunk) {
            if (isValidCustomer(customer)) {
                validChunk.add(customer);
            }
        }
        
        // Save chunk to database
        customerRepository.saveAll(validChunk);
        
    } catch (Exception e) {
        // This chunk failed, but continue with next chunk
        log.error("Failed to process chunk: {}", e.getMessage());
        // Continue processing other chunks
    }
}

PATTERN 2: Try-Catch at Multiple Levels
----------------------------------------
Layer 1: Controller Level
  Catches and returns HTTP error responses

Layer 2: Service Level
  Catches and logs business logic errors

Layer 3: Adapter Level
  Catches and handles parsing errors

Layer 4: Repository Level
  Catches and handles database errors

@PostMapping("/ingest")
public ResponseEntity<Map<String, Object>> ingestDataset(@RequestBody Map<String, String> request) {
    try {
        String directoryPath = request.get("directory_path");
        Map<String, Object> result = facade.ingestDatasets(directoryPath);
        return ResponseEntity.ok(result);

    }
}

private static LocalDateTime parseTimestamp(String value) {
    if (value == null || value.trim().isEmpty()) {
        return null;  // ← Safe fallback
    }
    try {
        return LocalDateTime.parse(value, TIMESTAMP_FORMATTER);
    } catch (Exception e) {
        return null;  // ← Don't crash, return null
    }
}

private static Integer parseInteger(String value) {
    if (value == null || value.trim().isEmpty()) {
        return null;
    }
    try {
        return Integer.parseInt(value);
    } catch (NumberFormatException e) {
        return null;  // ← Safe fallback
    }
}
